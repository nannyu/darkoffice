# 事件库规划

## 1. 文档目的

本文档定义《暗黑职场》第一版事件库的结构规划与最小内容要求。

## 2. 事件库目标规模

第一版建议：

- 完整目标：40 到 60 张事件卡
- 原型可玩下限：20 到 30 张事件卡

## 3. 事件库组织方式

建议双重组织：

1. 按事件类型组织
2. 按来源角色组织
3. 按是否触发掷骰组织

## 4. 事件类型清单

- 工作任务类
- 甩锅类
- 情绪压榨类
- 政治站队类
- 灰色合规类
- 甲方需求类
- 绩效制度类
- 恢复机会类

## 5. 事件种子示例

建议优先补出的典型事件：

- 今晚先把新版方案出掉
- 你顺手帮我处理一下
- 这是谁负责的
- 先倒签一下
- 你支持哪个方案
- 明天老板要看
- 需求先做了再确认
- 绩效沟通安排一下
- 先别发邮件
- 午休也能简单碰一下

## 6. 掷骰触发分层

### 6.1 默认确定性事件

以下事件默认不触发掷骰，重点用于稳定推进节奏：

- 普通任务分配
- 常规跟进
- 低压沟通
- 日常恢复机会

### 6.2 必触发掷骰事件

以下事件通常属于 `critical_event`，应直接触发 `2d6 + 修正值`：

- 今晚先把新版方案出掉
- 这是谁负责的
- 先倒签一下
- 你支持哪个方案
- 绩效沟通安排一下

### 6.3 条件触发掷骰事件

以下事件建议用 `conditional`：

- 明天老板要看
  说明：在普通工作时可以确定性推进；若叠加深夜、低 EN、项目截止或高压角色，则触发掷骰
- 需求先做了再确认
  说明：若玩家已有留痕或需求确认准备，可按确定性结算；否则在高风险条件下触发掷骰
- 先别发邮件
  说明：若用户选择对抗、留痕或拒绝时触发掷骰

## 7. 推荐首批事件清单与判定映射

| 事件 | 类型 | 默认判定 | 说明 |
| --- | --- | --- | --- |
| 今晚先把新版方案出掉 | 工作任务 / 加急 | `required` | 高压上司场景 |
| 你顺手帮我处理一下 | 推活 / 模糊责任 | `conditional` | 若涉及拒绝、转移或留痕对抗时触发 |
| 这是谁负责的 | 甩锅 / 问责 | `required` | 典型关键追责事件 |
| 先倒签一下 | 灰色合规 | `required` | 高风险流程事件 |
| 你支持哪个方案 | 政治站队 | `required` | 高风险组织政治事件 |
| 明天老板要看 | 汇报 / 加压 | `conditional` | 看当前状态与时间段 |
| 需求先做了再确认 | 甲方需求 / 边界模糊 | `conditional` | 看是否已有准备动作 |
| 绩效沟通安排一下 | 绩效制度 | `required` | 典型关键节点事件 |
| 先别发邮件 | 留痕对抗 | `conditional` | 看用户是否坚持留痕 |
| 午休也能简单碰一下 | 恢复机会打断 | `none` | 主要用于节奏与资源消耗 |
## 8. 事件库建设要求

- 每类事件都应有明确现实映射
- 同类事件应体现不同强度，而不是只是换名字
- 事件必须明确会生成什么后果
- 事件应能与角色被动形成稳定联动
- 会触发掷骰的事件必须明确是 `required` 还是 `conditional`
- 会触发掷骰的事件必须说明修正值重点来自哪些因素
