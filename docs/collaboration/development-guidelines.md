# 项目开发规范

## 1. 文档目的与适用范围

本文档是《暗黑职场》项目当前阶段的开发与协作总规范，适用于：

- 项目负责人
- 未来的人类协作者
- 参与项目推进的 AI/Agent
- 后续任何实现、验证、重构与文档迁移工作

本文档解决的是“如何有秩序地推进项目”，而不是“当前具体采用哪种技术栈实现”。

## 2. 当前项目阶段与开发原则

### 2.1 当前阶段定义

当前阶段定义为：文档先行、规则先行、结构先行。

在系统规则、内容边界和协作方式未形成稳定共识前，不启动正式开发，不进行技术栈绑定，不进行大规模目录落地。

### 2.2 核心开发原则

1. 先文档，后实现。
2. 先定义边界，后扩展细节。
3. 先最小闭环，后丰富内容。
4. 先单一事实源，后多处引用。
5. 先验证规则，后堆砌功能。
6. 先保守假设，后技术定案。

### 2.3 当前阶段允许的工作

- 搭建文档体系
- 明确系统边界与术语
- 梳理玩法规则与数据结构草案
- 制定协作规范、变更流程与验证原则
- 设计未来实现所需的目录与模板

### 2.4 当前阶段禁止的工作

- 在技术栈未定前直接搭建正式工程骨架
- 将未验证的系统规则写死为实现约束
- 通过原型代码替代策划与系统共识
- 在多个文档中平行定义同一规则

## 3. 单一事实源与文档优先级

### 3.1 单一事实源原则

每类信息只能有一个正式归属位置。任何新共识都应写入对应事实源，而不是散落在聊天记录、临时草稿或重复文档中。

### 3.2 当前文档优先级

1. `docs/project/`：项目定位、范围、阶段目标、术语的事实源
2. `docs/design/`：玩法体验与整体设计的事实源
3. `docs/systems/`：系统规则、流程和参数定义的事实源
4. `docs/content/`：内容模板、角色池、事件池和文本规范的事实源
5. `docs/collaboration/development-guidelines.md`：开发与协作规范事实源
6. `docs/INDEX.md`：导航入口与阅读顺序事实源
7. 根目录历史文档：历史参考，不作为当前最高优先级依据

### 3.3 冲突处理规则

- 当结构化子文档与历史总案冲突时，以结构化子文档为准。
- 当导航文档与正式规范冲突时，以正式规范为准。
- 当多个子文档发生冲突时，必须新增一次文档修订，消除冲突后再继续推进。
- 未写入事实源的口头结论，默认不视为项目共识。

## 4. 仓库结构与命名规范

### 4.1 根目录规则

- 根目录只保留入口级文件与少量历史文档
- 所有正式文档统一进入 `docs/`
- 不在根目录堆放大量临时文档、讨论稿和版本副本

### 4.2 文档目录规则

```text
docs/
├── project/
├── design/
├── systems/
├── content/
├── collaboration/
└── archive/
```

各目录含义：

- `project/`：项目层定义
- `design/`：玩法与体验设计
- `systems/`：可执行规则
- `content/`：内容资产与模板
- `collaboration/`：流程、规范、交接
- `archive/`：历史资料与迁移说明

### 4.3 文件命名规则

- 正式文档文件名优先使用英文 `kebab-case`
- 文档标题可以使用中文
- 避免在正式事实源中持续新增中文文件名、空格文件名和版本号后缀文件名
- 不使用 `final`、`最新版`、`修订版2` 这类不可维护命名

推荐示例：

- `core-loop.md`
- `card-system.md`
- `event-generation.md`
- `agent-handoff.md`

不推荐示例：

- `新版本最终版.md`
- `系统文档 真的最终版.md`
- `卡牌系统V1.2-修正版.md`

## 5. 任务拆分与迭代节奏

### 5.1 任务拆分原则

每次推进只解决一个清晰问题。任务必须可以明确回答以下问题：

- 本次改动解决什么问题
- 影响哪些文档或模块
- 产出是什么
- 如何验证已完成

### 5.2 推荐任务粒度

推荐将任务控制在以下粒度：

- 一次只完善一个主题文档
- 一次只迁移一个事实源
- 一次只解决一类冲突或缺口
- 一次只做一个闭环，不并行推进多个模糊目标

### 5.3 当前阶段的迭代节奏

建议按以下节奏推进：

1. 搭建文档骨架
2. 明确事实源
3. 迁移历史内容
4. 去重与消歧
5. 补充缺失主题
6. 完成一轮自检
7. 再决定是否进入技术预研

## 6. 变更流程与提交要求

### 6.1 变更前要求

任何开发或设计动作前，必须先确认：

- 当前变更对应的事实源文档在哪里
- 是否会与已有文档冲突
- 是否需要先补文档再继续

### 6.2 文档阶段的标准变更流程

1. 确认目标与边界
2. 找到对应事实源
3. 更新事实源文档
4. 更新索引或迁移说明
5. 做一致性检查
6. 记录本次变更影响

### 6.3 提交信息要求

未来启用版本控制后，提交信息应遵循：

- 说明本次变更主题
- 说明变更类型
- 避免使用无意义提交说明

推荐格式：

- `docs: define development guidelines`
- `docs: split card system into systems section`
- `docs: migrate legacy planning references`

不推荐格式：

- `update`
- `fix`
- `again`

## 7. 文档更新规范

### 7.1 更新规则

- 修改规则时，必须直接修改对应事实源
- 修改事实源后，必要时同步更新 `docs/INDEX.md`
- 不允许只改摘要，不改正式定义
- 不允许在多个地方复制同一段规则后分别演化

### 7.2 文档写作要求

- 优先写结论，再写解释
- 使用稳定术语，避免同义词乱跳
- 规则、流程、字段尽量结构化表达
- 说明边界时，要明确“包含什么”和“不包含什么”
- 技术未定项必须明确标注“待定”，不能伪装成已定方案

### 7.3 文档成熟度标记

建议后续为核心文档增加成熟度状态：

- `draft`：草稿，允许较大调整
- `reviewing`：评审中，尽量避免结构性改写
- `stable`：稳定，可作为实现依据
- `archived`：归档，不再继续维护

## 8. 测试与验证原则

### 8.1 总原则

在技术栈未定前，验证优先于实现。任何结论都应尽量满足“可检查、可复述、可追踪”。

### 8.2 文档阶段的验证方式

- 检查是否存在重复定义
- 检查是否存在术语冲突
- 检查章节是否覆盖当前目标
- 检查规则是否自洽
- 检查是否存在未标注的假设

### 8.3 进入开发阶段后的验证底线

虽然当前不进入开发，但先设定底线：

- 新功能必须有明确验收标准
- 修复性改动必须有复现路径与验证结果
- 任何“已完成”声明都应有可验证证据支撑
- 文档、规则、实现三者出现偏差时，必须回到事实源对齐

## 9. 技术实现预留规范

### 9.1 当前状态

当前项目技术栈未定，以下内容只定义原则，不定义具体框架或语言。

### 9.2 未来技术定案前必须回答的问题

- 项目首个可运行原型是什么形态
- 是纯文本原型、桌面交互原型，还是 Web 原型
- 数据以静态文件驱动，还是引入服务端状态管理
- 内容生产与系统运行是否共享一套数据结构
- AI/Agent 在运行态中扮演什么角色

### 9.3 技术实现通用约束

- 先围绕最小闭环设计目录，而不是为未来所有可能性过度预埋
- 模块边界必须能映射回文档中的系统边界
- 规则数据应尽量与表现层解耦
- 可配置内容优先结构化，不硬编码在表现逻辑中
- 临时技术决策必须记录理由和影响范围

## 10. AI/Agent 协作规范

### 10.1 开始任务前的最小读取集

AI/Agent 在开始任何任务前，至少应读取：

1. `README.md`
2. `docs/INDEX.md`
3. 本次任务对应的事实源文档
4. `docs/collaboration/development-guidelines.md`

### 10.2 AI/Agent 工作要求

- 先确认当前任务属于哪个文档域
- 先找事实源，再做新增内容
- 不根据单一历史文档直接扩写全局规则
- 做出假设时必须显式说明
- 发现冲突时先停下并回到文档层处理

### 10.3 AI/Agent 交接要求

交接内容至少包括：

- 本次修改了什么
- 修改依据是什么
- 哪些问题已经解决
- 哪些问题仍待确认
- 下一步建议看哪份文档

## 11. 禁止事项

- 禁止绕过文档直接定义项目核心规则
- 禁止多个文档重复定义同一事实且长期不清理
- 禁止把历史草稿当作当前最高优先级依据
- 禁止在技术栈未定时伪造工程约束
- 禁止为了“先做起来”而跳过边界、术语和验收定义
- 禁止把聊天结论当作正式共识却不落文档
